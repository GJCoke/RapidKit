# 架构设计

该项目采用「Monorepo 全栈一体化架构」，整合前端、后端、桌面端三大应用形态和多个共享功能模块，核心目标是实现「代码复用最大化、协作效率最优化、部署流程标准化」，以下从架构核心、设计原则、关键特性、技术链路等方面详细说明：

## 一、架构核心：Monorepo 单仓库组织模式

::: warning 提示
本文概述了项目基于 Monorepo 的整体架构，包括组织模式、设计原则、构建链路、发布机制及未来扩展方向，旨在提供统一的技术认知与团队协作规范。
:::

### 核心思想

打破传统多仓库（前端、后端、文档分离）的壁垒，将所有相关项目（应用、库、文档）集中管理在一个代码仓库中，通过「工作区（Workspaces）」机制实现依赖隔离与共享，解决多仓库模式下的依赖不一致、代码复用困难、版本管理混乱等问题。

### 组织逻辑分层

1. **应用层（apps/）**：面向最终用户的可执行产品，包括前端 SPA、后端 API 服务、Electron 桌面客户端。每个应用独立部署、独立运行，但共享底层配置和工具链，避免重复搭建项目环境。
2. **共享层（packages/）**：支撑应用层的通用功能模块，封装可复用的工具、组件、逻辑，通过包管理器（如 pnpm）以「内部依赖」的形式被应用层引用，确保功能一致性和维护成本最低化。
3. **配置层（根目录）**：统一管理全仓库的工具链配置（ESLint、Prettier、TypeScript）、Git 钩子配置（Husky）、任务编排配置（Turborepo），实现「一处配置，全局生效」，避免团队成员因本地配置差异导致的协作冲突。
4. **辅助层（docs/、scripts/）**：提供文档支持和运维辅助，确保项目的可维护性和可扩展性，降低团队沟通成本和操作成本。

## 二、设计原则

### 1. 模块化与边界隔离

- **应用隔离**：apps/ 下的每个应用（frontend/backend/desktop）均为独立的运行单元，拥有自己的 package.json、入口文件和依赖清单，可单独启动、构建、部署，互不干扰。例如：后端服务的重构升级不会直接影响前端应用的运行，前端的版本迭代也无需改动后端代码。
- **共享封装**：packages/ 下的每个库均遵循「单一职责原则」，专注于某一类功能（如 HTTP 请求、工具函数、颜色管理），通过明确的接口暴露功能，隐藏内部实现细节。应用层仅需依赖库的接口，无需关心内部逻辑，便于后续升级和替换。
- **依赖清晰**：通过 Monorepo 工作区机制，明确应用与共享库、共享库之间的依赖关系，避免循环依赖，同时借助 Turborepo 的依赖图分析，确保构建和运行时的依赖正确性。

### 2. 标准化与一致性

- **编码规范统一**：通过 ESLint + Prettier 强制统一代码风格和语法规则，支持自动修复常见问题，减少代码评审中的格式争议；TypeScript 全局配置确保类型定义一致，提升代码可读性和可维护性，降低运行时错误。
- **提交规范统一**：通过 Husky + commitlint 强制校验 Git 提交信息，遵循 Conventional Commits 规范，确保提交信息结构化（如 `feat: add user login`、`fix: fix form active bug`），便于自动化生成变更日志和版本管理。
- **工具链统一**：全仓库使用统一的构建工具（Turborepo 编排、Rollup/Vite 打包）、包管理器（pnpm）、测试工具，避免因工具差异导致的构建失败、依赖冲突等问题，降低团队成员的学习成本。

### 3. 高效构建与缓存优化

- **任务编排智能化**：Turborepo 作为核心构建工具，通过 turbo.json 定义任务依赖（如 frontend:build 依赖 packages:build），自动分析依赖图，并行执行无依赖的任务（如同时构建 backend 和 frontend），大幅提升构建效率。
- **缓存机制最大化**：Turborepo 支持缓存任务结果（如构建产物、lint 结果、测试报告），仅当代码或配置发生变更时才重新执行任务，重复构建时可直接复用缓存，尤其在 CI/CD 流水线中能显著缩短构建时间（如从 30 分钟压缩至 5 分钟）。
- **增量构建支持**：各应用和共享库支持增量构建，仅重新编译变更的文件，而非全量构建，进一步提升开发阶段的热更新速度和生产环境的构建效率。

### 4. 可扩展性与跨平台支持

- **技术栈兼容扩展**：前端采用 Vue 3 生态，支持组件化、自定义 Hook 等扩展能力；后端采用 FastAPI，支持异步扩展、中间件机制，可灵活集成数据库（MySQL、MongoDB）、缓存（Redis）等服务；桌面端基于 Electron，支持原生功能扩展（如集成第三方 SDK）。
- **跨平台复用**：前端代码可直接复用于 Electron 桌面端（仅需适配少量桌面端特有 API），实现「一套代码，多端部署」，减少跨平台开发的重复工作量；共享库（如 utils、hooks）可同时被前端、桌面端、后端（Node.js 环境）引用，最大化代码复用率。
- **业务模块扩展**：Monorepo 架构支持快速新增应用或共享库，例如未来可新增移动端应用（React Native）、小程序应用，只需在 apps/ 目录下新增子项目，即可复用现有共享库和配置，无需重新搭建整套基础设施。

## 三、关键技术链路

### 1. 开发链路

开发者克隆仓库后，通过根目录 `pnpm install` 安装所有依赖，执行 `pnpm dev:xxx` 启动对应应用的开发服务；开发过程中，代码变更会触发热更新（前端 Vite 热更新、后端 FastAPI 自动重载），同时 Husky 钩子在提交代码时自动执行 lint 和测试，确保提交质量。

### 2. 构建链路

执行 `pnpm build` 时，Turborepo 按依赖顺序执行构建任务：先构建 packages/ 下的共享库（如 axios、utils），再构建 apps/ 下的应用（frontend、backend、desktop）；每个应用的构建产物按统一规范输出（如 frontend 构建至 dist/ 目录，backend 打包为 Python Wheel 包），构建结果可缓存至本地或远程（如 CI 缓存服务）。

### 3. 发布链路

通过 Changesets 工具管理版本发布：开发者提交代码时，同步提交变更集（.changeset/ 下的 Markdown 文件）；发布前执行 `pnpm changeset version`，工具自动计算各包新版本号、更新 package.json 和 CHANGELOG.md；执行 `pnpm publish` 可批量发布所有变更的包和应用，配合 CI/CD 流水线可实现自动化发布（如合并至 main 分支后自动构建并发布）。

### 4. 部署链路

- 前端应用：构建产物（静态资源）部署至 CDN 或静态文件服务器（如 Nginx），通过环境变量配置后端 API 地址。
- 后端服务：打包为 Python 应用，部署至云服务器（如 AWS EC2、阿里云 ECS）或容器化（Docker）部署至 Kubernetes 集群，暴露 API 端口供前端和桌面端调用。
- 桌面客户端：通过 Electron Builder 打包为各平台安装包（.exe、.dmg、.deb），用户下载安装后，通过配置文件连接后端 API。

## 四、架构优势与价值

### 1. 团队协作效率提升

- 无需跨仓库切换：开发前后端、桌面端功能时，无需在多个仓库间切换，代码查找、修改、提交更便捷。
- 依赖同步更新：共享库升级后，所有引用该库的应用可同步获取更新，无需手动维护多仓库的依赖版本。
- 新人上手成本低：统一的配置、规范和文档，新人可快速搭建开发环境，熟悉项目结构，无需适配不同仓库的差异化配置。

### 2. 代码质量与可维护性提升

- 统一规范落地：通过工具链强制执行编码规范和提交规范，减少「坏代码」和「不规范提交」，提升代码整体质量。
- 共享逻辑集中维护：核心工具函数、业务逻辑封装在共享库中，修改时只需改一处，所有应用同步生效，避免重复修改导致的不一致。
- 问题追溯便捷：所有代码变更集中在一个仓库，可通过 Git 日志快速追溯某个功能或 Bug 的修改记录，便于代码评审和问题排查。

### 3. 部署与运维效率提升

- 统一构建流程：全仓库使用一套构建脚本和配置，避免多仓库构建流程不一致导致的部署失败。
- 缓存加速 CI/CD：Turborepo 缓存机制大幅缩短 CI/CD 流水线执行时间，提升迭代发布速度（如支持一天多次发布）。
- 跨平台部署简化：前端和桌面端复用代码，部署时可同步发布，减少跨平台部署的协调成本和工作量。

## 五、适用场景与未来扩展

该架构适用于「多端协同、功能复用性高、团队协作紧密」的全栈项目，尤其适合需要同时维护 Web 端、桌面端、后端服务的团队。未来可扩展方向包括：

- 新增移动端应用（React Native/Flutter），复用现有 utils、hooks、API 层代码；
- 引入微前端架构，将 frontend 拆分为多个子应用，进一步提升前端模块化程度；
- 强化监控与日志系统，集成 Prometheus、Grafana 等工具，实现全链路监控；
- 引入 Serverless 部署模式，将 backend 拆分为多个 Serverless 函数，降低运维成本。

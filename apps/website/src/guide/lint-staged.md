## lint-staged 配置与使用指南

本指南介绍如何在项目中使用 lint-staged 针对当前暂存区的文件执行检查，从而减少无关文件的处理，提高代码提交效率。

## 什么是 lint-staged？

lint-staged 是一个专为 Git 暂存区文件设计的工具。  
它允许你在执行提交前，仅对本次提交涉及的文件运行代码检查或格式化工具，而不是对整个项目进行扫描。

其主要优点包括：

- 明显提升检查效率
- 避免提交未格式化或未修复的问题文件
- 与 Husky 搭配实现自动化提交流程
- 支持灵活的任务配置

## lint-staged 的作用场景

lint-staged 通常用于以下场景：

- 提交前自动运行 ESLint
- 使用 Prettier 格式化暂存文件
- 对样式文件执行 StyleLint
- 对 Markdown、JSON 等文件执行格式化
- 避免提交未清理的调试代码或语法错误

由于它只处理 Git 暂存区文件，因此速度远快于全量运行格式化或 Lint。

## 常见配置内容说明

lint-staged 的配置文件通常包含以下内容：

### 1. 匹配文件模式

用于指定哪些文件会触发检查，例如 JavaScript、TypeScript、Python、CSS、JSON 等。

### 2. 设置任务列表

可以对不同类型的文件设置不同的任务，例如：

- 脚本文件执行 Lint
- 所有代码运行 Prettier 格式化
- 特定语言执行类型检查

lint-staged 会自动将最终结果重新添加到暂存区。

## 运行机制说明

提交过程中的运行流程如下：

1. 执行 Git commit
2. Husky 的 pre-commit 钩子触发
3. lint-staged 获取当前暂存区文件
4. 按配置规则对文件执行检查
5. 将执行后的文件重新加入暂存区
6. 若所有任务通过，则继续提交
7. 若某个任务失败，提交被阻止

此机制有效保证提交的代码始终符合格式与质量要求。

## 与 Husky 的集成

lint-staged 最常见的使用方式是配合 Husky 的 pre-commit 钩子。  
Husky 负责触发 lint-staged，而 lint-staged 专注于处理暂存区文件。

推荐将所有提交前的检查移到 lint-staged 中，以保持流程统一和高效。

## 常见问题排查

### 1. lint-staged 没有生效

常见原因包括：

- Husky 未正确安装
- pre-commit 钩子未执行
- 文件未加入暂存区
- lint-staged 配置文件命名错误

### 2. 格式化后的文件未加入提交

lint-staged 默认会自动重新添加文件，但如果任务中执行了不会修改文件的操作，需要确保未跳过文件添加步骤。

### 3. 提交速度过慢

通常原因是匹配模式过大或执行了全量检查，可以通过以下方式优化：

- 精简任务
- 减少文件匹配范围
- 将耗时任务放到 pre-push 执行

## 最佳实践建议

为获得最佳使用体验，推荐遵循以下实践：

- 所有提交前的检查统一通过 lint-staged 执行
- 将类型检查等耗时任务移动到 pre-push
- 对不同文件类型设置不同任务，提高效率
- 配合 Prettier 和 ESLint 等工具统一格式
- 避免对 node_modules 或大型目录设置匹配规则

## 总结

lint-staged 是一个高效、灵活的工具，用于在提交前检查暂存区文件。  
它能显著提升提交效率、提高代码质量，并避免无关文件的格式化和 Lint，从而构建更高效、更可靠的开发流程。

通常情况下，将 lint-staged 与 Husky、ESLint、Prettier 等工具结合使用，可以形成一套完整的提交前自动化校验体系，为团队提供一致而高效的开发体验。

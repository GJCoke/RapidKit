## 概述

Changesets 是一款专为 Monorepo 架构设计的版本管理与变更日志生成工具，核心作用是自动化管理多包版本迭代、统一生成变更日志（CHANGELOG.md），解决手动维护版本号易出错、变更记录不规范的问题。本项目中，Changesets 负责管控 packages/ 下所有共享库及 apps/ 下应用的版本发布流程，确保版本语义化、变更记录可追溯。

## 配置步骤

### 核心配置文件说明

默认配置已适配本项目 Monorepo 结构，关键字段含义如下：

```jsonc
{
  "$schema": "https://unpkg.com/@changesets/config@3.0.0/schema.json",
  "changelog": "@changesets/cli/changelog", // 变更日志生成器（默认）
  "commit": false, // 是否自动提交版本更新的变更（建议关闭，手动提交更可控）
  "fixed": [], // 固定版本的包（多包强依赖时使用，如 ["@xxx/axios", "@xxx/utils"]）
  "linked": [], // 版本联动的包（一个包更新，其他包同步更新版本）
  "access": "public", // 发布范围：public（公包）/restricted（私有包）
  "baseBranch": "main", // 基准分支（用于对比变更）
  "updateInternalDependencies": "patch", // 内部依赖更新策略：patch/minor/major
  "ignore": [], // 忽略版本管理的包（如纯配置包）
  "privatePackages": {
    "version": "patch", // 私有包版本更新默认策略
  },
}
```

### 适配项目的自定义配置

根据本项目特点，推荐调整以下配置以优化使用体验：

1. 自定义变更日志模板（可选）：若默认 CHANGELOG 格式不满足需求，可指定自定义日志生成函数路径，通过编写脚本实现 getReleaseLine 方法，自定义日志内容格式、排版样式等；
2. 忽略无需版本管理的目录：若 apps/backend（Python 服务）无需 Changesets 管控版本，可将其添加至 ignore 配置项中，避免不必要的版本处理；
3. 配置版本联动：若部分共享库存在强依赖关系（如 @xxx/axios 与 @xxx/utils），更新其一需同步更新版本以保证兼容性，可在 linked 配置中添加关联组合。

## 使用流程

### 核心概念

- 变更集（Changeset）：描述单个代码变更的类型（feat/fix/docs 等）、影响的包、变更说明，以 Markdown 文件存储在 .changeset/ 目录，是版本更新的核心依据；
- 语义化版本（SemVer）：版本号格式为 主版本号.次版本号.补丁号（如 1.2.3），对应不同变更级别：
  - major：不兼容的 API 变更（主版本），如删除核心功能、修改接口参数格式；
  - minor：新增功能且兼容（次版本），如添加新方法、扩展现有功能；
  - patch：修复 Bug 且兼容（补丁版本），如修复逻辑错误、优化性能问题。

### 日常开发 - 创建变更集

每次提交代码时，若变更涉及 packages/ 或需版本更新的 apps/ 子包，需按以下步骤创建变更集：

1. 执行启动命令`pnpm changeset`，触发交互式配置流程；
2. 选择变更影响的包：工具会列出所有可管控的包，按空格选中本次变更涉及的包（支持多选），按回车确认；
3. 选择版本变更类型：针对每个选中的包，根据变更内容选择 major/minor/patch；
4. 填写变更说明：遵循 Conventional Commits 规范，输入简洁准确的变更描述（如 feat: 新增请求超时重试机制），支持多行输入，完成后按对应快捷键确认；
5. 生成变更文件：.changeset/ 目录下会生成随机命名的 .md 文件，记录本次变更的包、版本类型及说明；
6. 提交变更文件：将生成的变更文件与代码变更一起提交至 Git 仓库，确保变更记录可追溯。

### 版本发布 - 应用变更集

当需要发布新版本时，按以下步骤执行，完成版本更新与发布：

#### 步骤1：更新版本号与 CHANGELOG

执行命令`pnpm changeset version`，工具会自动完成一系列操作：

1. 分析所有未处理的变更集，计算各包的新版本号；
2. 更新对应包的 package.json 中的 version 字段；
3. 为每个包生成/更新 CHANGELOG.md 文件，追加本次变更记录；
4. 删除 .changeset/ 下已处理的变更集文件；
5. 若配置了 linked 包，同步更新联动包的版本号。

#### 步骤2：确认版本更新内容

手动检查各包的 package.json 版本号是否符合预期，查看 CHANGELOG.md 内容是否完整、描述准确，若存在问题可手动调整后重新提交。

#### 步骤3：发布包（可选，针对 npm 发布的包）

若 packages/ 下的库需发布至 npm 仓库，需先确保已登录 npm 账号，且包名未被占用，可选择批量发布所有变更的包，或进入具体包目录单独发布。

#### 步骤4：提交版本更新记录

将版本号、CHANGELOG 的变更提交至 Git 仓库，并按规范打 Tag，Tag 格式为 v{版本号}（如 v1.2.3），确保版本可追溯。

### 特殊场景处理

#### 场景1：临时跳过变更集（仅本地测试）

若开发过程中无需记录变更（如调试代码、临时修改），可创建空变更集，避免影响版本管理流程。

#### 场景2：撤销未提交的变更集

若生成变更集后发现错误（如选错包、版本类型有误），直接删除 .changeset/ 下对应的 MD 文件即可。

#### 场景3：批量处理变更集

若多个变更集未处理，执行版本更新命令会一次性应用所有变更集，自动汇总生成 CHANGELOG，无需逐个处理。

#### 场景4：私有包版本管理

若项目包含私有包（无需发布至 npm），只需执行版本更新命令更新版本号和 CHANGELOG，跳过发布步骤即可。

## 团队协作规范

### 变更集创建规范

1. 仅对有实际功能/代码变更的内容创建变更集，纯格式调整、注释修改、日志清理等无需创建；
2. 变更描述需清晰规范：前缀明确（feat/fix/docs/refactor/perf 等），内容具体，避免模糊描述（如“优化代码”），需说明具体变更点（如“fix: 修复 axios 拦截器未捕获网络错误的问题”）；
3. 版本类型选择准确：严格按照语义化版本规则选择，避免随意升级版本级别导致兼容性问题。

### 发布流程规范

1. 版本发布需在 main 分支执行，禁止在开发分支、特性分支直接发布；
2. 发布前需通过 CI 校验，确保代码 lint 无错误、单元测试全部通过、构建流程正常；
3. 每次发布后必须打 Tag，Tag 命名遵循 v{版本号} 格式，便于版本回溯；
4. 发布后需同步更新 docs/website 中的版本说明文档，记录主要变更内容、兼容说明等。

## 常见问题排查

### 执行版本更新命令无反应

检查 .changeset/config.json 的 ignore 配置，确认目标包未被忽略；检查 .changeset/ 下是否有未处理的变更集文件；确认包的 package.json 中 name 字段与变更集中的包名一致。

### CHANGELOG 未生成

检查 config.json 中 changelog 配置是否正确，若使用自定义生成器需确保函数无语法错误；确认变更集描述不为空，空描述会导致 CHANGELOG 无对应内容。

### 版本号更新不符合预期

检查变更集中的版本类型（major/minor/patch）是否选择正确；检查 linked/fixed 配置，确认联动包版本逻辑符合预期；确认 updateInternalDependencies 配置是否影响内部依赖的版本更新规则。

### 发布时提示权限不足

确认 npm 账号已登录，且拥有该包的发布权限；若为私有包，检查 config.json 中 access 配置是否为 restricted。

## 扩展能力

### 集成 CI/CD

可在 GitLab CI/GitHub Actions 中集成 Changesets，实现自动化发布流程：设置触发条件为合并 PR 至 main 分支且包含变更集文件；

定义执行步骤为安装依赖 → 执行版本更新命令 → 提交版本更新 → 打 Tag → 执行发布命令，减少手动操作成本。

### 自定义变更日志

通过安装 @changesets/changelog-github 插件，可生成关联 GitHub PR 的 CHANGELOG，包含 PR 链接、贡献者信息等，提升变更记录的完整性。

安装后修改 config.json 中的 changelog 配置，指定插件及仓库地址即可生效。
